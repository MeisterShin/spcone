# 동선(게이트) 설계 (P3 확장 · 4/4, 마지막 항목)

## 배경
합의된 순서(통계 대시보드→수행원 관리→좌석배치→동선) 중 마지막 항목. 사용자 요구사항(그대로 인용): "동선은 행사장별로 게이트가 추가로 설정할 수 있도록 하면 될 것 같아. 이를테면 주차장 입장, 건물입장, 행사장입장 등으로 해서 체크인 버튼을 세분화 할 수 있도록 설계하는거지. 단일화해서 체크인만 확인하는 모드와 세부 설정에 따라 확인하는 모드를 나눠서 동작하게 하고." 이어서: "게이트 세개를 무조건 체크해야 하는건 아니고 최종 체크인만 되면 되는데 중간 상태값도 입력을 받게 해달라는거야." — 중간 게이트는 필수가 아닌 보조 기록이고, 최종 도착 처리(기존 체크인)만 실제로 중요함.

## 목표
행사별로 동선 게이트(예: 주차장 입장·건물 입장·행사장 입장)를 자유롭게 등록하고, 현장 체크인 화면에서 게이트별로 통과 여부를 개별 기록할 수 있게 한다. 게이트 등록이 없는 행사는 지금과 완전히 동일하게 동작한다(단일화 모드). 새로운 도착 상태·데이터 저장소를 만들지 않고 기존 체크인 트랜잭션(`doCheckin`)·감사·오프라인 큐를 그대로 재사용한다.

## 사용자 확인 사항 (설계 확정)
- 게이트는 **필수 순차 관문이 아니다** — 어떤 게이트든 순서 상관없이 개별적으로 눌러 기록할 수 있다.
- **배열의 마지막 게이트 = 실제 도착(기존 체크인)과 동일하게 처리**한다. 마지막 게이트를 누르는 순간(다른 게이트를 안 눌렀어도) 기존 `doCheckin()`이 그대로 실행된다 — VVIP 알림·중복 확인 모달·오프라인 큐·감사로그 전부 지금과 동일.
- 나머지(마지막이 아닌) 게이트는 순수 참고용 기록이며, 통과 여부가 최종 체크인을 막거나 강제하지 않는다.

## 데이터 모델 (새 컬렉션 없음)
- `events`에 `gates:[{id,name}]` 추가. 빈 배열(또는 미설정) = 단일화 모드, 1개 이상 = 게이트 모드. 별도의 모드 플래그 없이 배열 유무로 자동 전환.
- `eventGuests`에 `gatesPassed:{[gateId]:{at,by}}` 추가(기본 없음 → 빈 객체로 취급).
- `checkinLogs`에 기존 `action` 값 목록에 `'gate_pass'`/`'gate_undo'` 추가(기존 로그 스키마 그대로, `gateId` 필드만 추가로 실림).

## 로직 (기존 함수 재사용 중심)
1. `addGate(list)`/`removeGate(list,idx)`/`sanitizeGates(list)` — 수행원 관리 때의 `addCompanion`/`removeCompanion`/`sanitizeCompanions`와 완전히 동일한 형태의 순수 함수(불변 배열 조작 + 저장 시 빈 이름 제외).
2. `toggleGatePass(egId,gateId)` — 신규 함수.
   - 이미 도착 처리된 내빈은 아무 동작 안 함(게이트 버튼 자체가 화면에서 사라짐).
   - 이미 통과한 게이트를 다시 누르면 `gatesPassed`에서 제거(취소) + `gate_undo` 감사 기록.
   - 아직 안 통과한 게이트를 누르면 `gatesPassed[gateId]={at,by}` 기록 + `checkinLogs`에 `gate_pass` 기록. **오프라인 처리(`isOnline()`/`S.queue`/`flushQueue`)는 기존 `doCheckin`과 완전히 동일한 코드 패턴을 그대로 사용**(신규 오프라인 로직 없음).
   - 방금 누른 게이트가 `event.gates` 배열의 **마지막 항목**이면, 그 즉시 기존 `doCheckin(egId)`를 호출한다(로직 복제 없이 그대로 재사용 — VVIP 확인 모달·알림·오프라인 큐·감사·실행취소 토스트 모두 지금과 동일하게 동작).

## 화면
- **행사 생성/수정 모달**: "게이트(동선)" 섹션 추가. 수행원 관리와 동일한 UI 패턴(임시 배열에 담아 이름 입력 행 + `+ 게이트 추가`/삭제, "저장" 시에만 `event.gates`에 반영).
- **현장 체크인 페이지(`renderCI`)**: 카드 하단을 조건 분기.
  - 게이트 없음(기존과 동일): 카드 전체가 버튼, 탭하면 `doCheckin` 그대로.
  - 게이트 있음 + 아직 미도착: 카드는 더 이상 통짜 버튼이 아니라(내부에 여러 버튼이 들어가야 하므로), 상단(아바타·이름·소속)은 그대로 두고 하단에 게이트별 작은 버튼(통과 시 강조 스타일)을 배치. 각 버튼 클릭 시 `toggleGatePass`.
  - 게이트 있음 + 이미 도착: 지금과 동일한 "✓ 도착 완료" 요약 표시(게이트 버튼 숨김).

## 시드 데이터
데모 행사에 게이트 3개(주차장 입장/건물 입장/행사장 입장)를 기본으로 넣어 배포 직후 바로 시연 가능하게 한다.

## 에러 처리
- 게이트 미설정 행사는 현재 동작과 100% 동일(회귀 없음이 최우선 요구사항).
- 도착 취소(`undoCheckin`)가 실행돼도 `gatesPassed` 기록은 지우지 않는다(데이터 삭제 대신 상태 변경 원칙 — 실제로 그 시점에 그 게이트를 통과했다는 사실 자체는 유효한 이력). 취소 후 다시 도착 처리하려면 마지막 게이트를 다시 토글(껐다 켜기)하거나 게이트가 없는 것처럼 다른 경로로 체크인하면 된다 — 별도의 특수 케이스 코드는 만들지 않는다.

## 범위 밖
- 영접 관리·대시보드·통계·결과보고 화면에는 게이트 통과 상세를 노출하지 않는다(요청 범위인 "체크인 세분화"에 집중).
- 게이트 순서 강제(사용자가 명시적으로 원치 않음)·게이트별 도착예정시각 등 부가 기능 없음.

- `addGate`/`removeGate`/`sanitizeGates` 순수 함수 단위테스트(수행원 테스트와 동일한 형태)
- `toggleGatePass`가 일반 게이트에서는 `arrivalStatus`를 바꾸지 않고 `gatesPassed`만 갱신하는지
- 마지막 게이트를 누르면 `doCheckin`이 실제로 실행되어 `arrivalStatus`가 `'arrived'`로 바뀌는지(다른 게이트를 하나도 안 눌러도 동작하는지 포함)
